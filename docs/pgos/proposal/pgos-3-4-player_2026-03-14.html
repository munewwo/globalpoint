1. このシチュエーションの概要

プレイヤーの課題

「いま何レベルで次のブラインドがいくらか確認したい。残り何人か知りたい。アドオンしたいけどディーラーを呼ばないといけない。席替えの通知を聞き逃した。」

PGOSでは、トーナメントの進行状況をリアルタイムでアプリに表示。ブラインドスケジュール、残り人数、平均スタック、賞金情報がいつでも確認できる。アドオンやリエントリーもアプリから申請でき、席替え通知はプッシュ通知で確実に受信。

3-3（来店時）との関係

3-3 のチェックイン完了後、プレイヤーはPlay-in 状態でテーブルに着席。本ドキュメントはそこからエリミネーション（またはトーナメント終了）までの体験を扱う。トーナメント結果の確定・ペイアウトは「3-6. プレイ終了・退店時」で定義。

2. プレイ中の状態モデル

3-2 で定義したトーナメントプレイヤー状態モデルのうち、プレイ中に関連する遷移:

┌─────────┐ 席替え ┌─────────┐
│ Play-in │───────────→│ Play-in │（新テーブル/シート）
└────┬────┘ └─────────┘
   │
   ├── バスト（Reg期間中）──→ リエントリー可 → 再び Play-in
   │
   ├── バスト（Reg終了後）──→ Play-out（最終順位確定）
   │
   └── 最後まで生存 ──→ トーナメント終了（3-6へ）
状態説明プレイヤーアプリの表示
Play-inテーブルに着席してプレイ中トーナメント情報画面（ブラインド、残り人数等）
Seat-out一時離席中（ブラインドは消費される）「離席中」バッジ。戻る際は通知なし
Play-outレジストレーション終了後のバスト。最終順位確定順位・獲得ポイント（該当時）の表示

3. ユーザーストーリー

ストーリー1: 「アプリでトーナメント進行をリアルタイム確認」

プレイヤーAはサンデー杯に参加中。ハンドの合間にアプリを開くと、現在レベル6 / ブラインド 300-600 / アンテ 75 / 残り42名 / 平均スタック 28,000 / 次のブレイクまで12分が一目で確認できる。賞金テーブルを見ると「現在42名 → インマネは18名」と表示。あと24人の脱落でインマネだと分かる。

ストーリー2: 「アドオン申請 → ディーラーがチップを配布」

プレイヤーBはスタックが減ってきた。ブレイク中にアプリで「アドオンする」をタップ。支払い方法（ウェブコイン）を確認して申請完了。ディーラーのFloorアプリに通知が届き、ブレイク終了時にアドオン分のチップが配布される。

ストーリー3: 「席替え通知を受信して移動」

プレイヤーCのテーブルが解散（テーブルブレイク）。アプリにプッシュ通知「テーブル5 → テーブル2 / シート3 に移動してください」が届く。アプリ画面にも新しいテーブル/シート番号が表示され、迷わず移動。

5. プレイ中の操作

5-1. アドオン申請（A2-3-03）

ID要件詳細優先度Ph
A2-3-03アドオン申請アドオン可能期間中にアプリから申請。①アドオン額・チップ額の確認 ②支払い方法の選択（事前決済のウェブコイン/カード or 現金を選択） ③申請送信 → スタッフ/ディーラーに通知。アドオンは通常ブレイク中に処理されるが、クラブ設定によりプレイ中も可能。アドオン可能期間外は申請ボタンが非活性Should1

5-2. リエントリー申請（A2-3-06）新規

バスト後、レジストレーション期間内であればアプリからリエントリーを申請できる。

ID要件詳細優先度Ph
A2-3-06リエントリー申請バスト後にアプリに「リエントリーする」ボタンを表示（レジストレーション期間中 かつ リエントリー上限未達の場合のみ）。①エントリー費・チップ額の確認 ②支払い方法の選択 ③申請送信 → 新しいテーブル/シートが割り当てられ Play-in に復帰。残りリエントリー回数を表示（B1-3-14 の上限設定に基づく）Should1

5-3. 飲食オーダー（A2-4-01）

ID要件詳細優先度Ph
A2-4-01飲食オーダーテーブルを離れずにアプリからフード・ドリンクを注文。メニュー表示 → 商品選択 → 注文確定 → テーブルに配達。支払いはデポジット引落し or 退店時精算Should3

9. 検討事項・リスク

項目内容対応方針
チップカウントの公開範囲プレイ中のチップカウントをどこまで公開するか。全員に公開すると戦略に影響ファイナルテーブルのみ公開（デフォルト）。クラブ設定で変更可能
アドオン申請の乱用頻繁なアドオン申請によるスタッフ負荷アドオン可能回数の制限（クラブ設定）。ブレイク中のみ受付（デフォルト）
通知の頻度プレイ中の過度な通知がプレイの妨げになる重要通知（席替え・バスト）のみプッシュ。情報系（ブレイク等）はアプリ内のみ（設定可能）
リアルタイム情報の遅延クロック同期の遅延により実際と異なる情報が表示される5秒以内の同期を目標。最終的にはクロック（PGOS Clock）が正
オフライン時の情報表示ネットワーク切断時にリアルタイム情報が更新されない最後に取得した情報を表示し「オフライン」バッジを付与。席替え通知は復帰後に受信

10. 非機能要件

項目要件備考
クロック同期PGOS Clock との時刻差5秒以内WebSocket / SSE による同期
残り人数更新エリミネーション記録から3秒以内に反映API / Push
席替え通知バランシング確定から5秒以内に受信プッシュ通知
アドオン申請応答申請から確認まで3秒以内サーバー処理
バッテリー消費リアルタイム同期でバッテリー消費が過大にならない設計同期間隔の最適化
