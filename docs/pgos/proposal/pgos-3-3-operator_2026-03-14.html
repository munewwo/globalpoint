1. このシチュエーションの概要

運営の課題

「受付が混雑して回しきれない。初来店のお客様のゲスト登録に時間がかかる。リングゲームのお客様までカウンターに並ばせるのは非効率。注意人物を見逃してしまう。」

PGOSでは、トーナメントはQRスキャンで瞬時にチェックイン。リングゲームは先にプレイヤーを着席させ、スタッフがテーブルに出向いて手続きする設計でカウンター混雑を解消。ゲスト登録も簡素化し、カウンターフリー設計でどの端末・どのスタッフからでも同じ操作が可能。

トーナメントとリングゲームの運用フローの違い

トーナメントリングゲーム
受付方式受付カウンターでチェックイン（カウンターフリーも可）テーブルサイド受付: プレイヤーが着席後、スタッフがテーブルに出向く
本人確認タイミングチェックイン時（受付カウンター）着席後（テーブルサイド）
配席システム自動抽選スタッフ案内 or プレイヤーが空席に着席
支払い・チップチェックイン時に未払い精算テーブルでバイイン→支払い→チップ配布

3. ユーザーストーリー

T ストーリー1: 「QRスキャンで瞬時に受付完了」

トーナメント開場時間。スタッフAはFloorアプリを手に受付に立つ。プレイヤーがQRコードを提示。スキャンすると画面に「会員: 田中太郎 / eKYC済 / 年齢: 30代 / エントリー済: サンデー杯（ウェブコイン決済完了）」と表示。アラートなし。「チェックイン」ボタンをタップ → 自動配席実行 →「テーブル3 / シート5」を口頭で案内。所要時間10秒。

共通 ストーリー2: 「初来店ゲストの受付」

初来店のプレイヤーがカウンターに。PGOSアカウントなし。スタッフBがFloorアプリで「ゲスト登録」を選択。名前と生年月日を入力するとQRコードが自動発行され、プレイヤーのスマートフォンに表示。年齢が確認され「20歳以上」のバッジが表示。そのままデイリーNLHにウォークインエントリー。現金でエントリー費を受領し、チェックイン完了。

R ストーリー3: 「テーブルに出向いて受付」修正

NLH 1-2で席が空いた。PGOS Managerが自動的にウェイティングリスト1番のプレイヤーにプッシュ通知を送信。プレイヤーが店舗に到着し、スタッフが案内した席に着席。スタッフCがFloorアプリを持ってテーブルに出向き、プレイヤーのQRをスキャン → 本人確認 → バイイン額 ¥10,000 を確認 → 現金受領 → チップ ¥10,000相当を配布。カウンターを経由せず、テーブルですべて完結。

5. 3-3T トーナメント — チェックイン運用

トーナメントではチェックインが必要。受付カウンターまたはスタッフ（カウンターフリー）でQRスキャン → エントリー確認 → 支払い確認 → チェックイン → 配席の流れで処理する。

スタッフによるチェックインフロー:

① プレイヤーがQRコード提示 → Floor / Manager でスキャン
↓
② 会員情報＋エントリー状況＋アラート表示
↓ アラートなし or 確認済み
③ 未払いがある場合 → 支払い処理（現金受領 / カード決済）

④ 「チェックイン」ボタン → 自動配席実行
↓
⑤-a 空席あり → プレイイン（テーブル/シートを口頭案内 + プレイヤーアプリに通知）
⑤-b 全席満席 → オルタネイト（番号案内 + オルタネイトリストに追加）

6. 3-3R リングゲーム — テーブルサイド受付修正

6-1. 設計思想: 着席優先 → テーブルサイド受付新規

リングゲームはカウンターを経由しない

リングゲームでは、プレイヤーはまず席に座り、その後スタッフがテーブルに出向いて手続きを行う。トーナメントのような受付カウンターでのチェックインは行わない。

  • プレイヤーが着席: 空席に座る or スタッフに案内された席に座る
  • スタッフがテーブルへ: Floorアプリを持ってテーブルに出向き、QRスキャン → 本人確認 → バイイン確認 → 支払い → チップ配布
  • 席QRによるセルフ受付: プレイヤーが席QR（B1-3-15）をスキャンして自分で受付を完了させることも可能（チップ配布のみスタッフが対応）
ID要件詳細優先度Ph
B2-6-01テーブルサイド受付リングゲームのプレイヤーが着席後、スタッフがFloorアプリでテーブルに出向き、QRスキャン → 本人確認 → バイイン額確認 → 支払い処理 → チップ配布までを一連で実行。テーブルの座席表上に「新着席（受付未処理）」を表示し、スタッフに通知。プレイヤーが席QRをスキャンした場合は、本人確認＋バイイン額選択までをセルフで完了し、「チップ配布待ち」としてスタッフに通知新規Must1

6-2. ウェイティング呼び出し → 着席 → テーブル処理

呼び出し後のフロー（運営側）:

① 席が空く → PGOS Managerが自動でウェイティング1番に通知送信
↓ 1分以内に応答
② プレイヤーが来店 → スタッフが席に案内（または自分で着席）
↓
③ Floorアプリに「新着席（受付未処理）」が表示される
↓
④ スタッフがテーブルに出向き → QRスキャン → 本人確認 → バイイン額確認
↓
⑤ 支払い処理 → チップ配布 → 座席表を「プレイイン」に更新

呼び出しルールの詳細（1分/3回/保留リスト）は3-2で定義済み（B2-2-03, B2-2-04, B2-2-06）。1分ルールの対象はプレイヤーアプリ上の応答操作であり、着席後のテーブル受付処理とは別のタイムライン。

6-3. 直接来店 → 着席 → テーブル処理

空席がある場合のフロー（運営側）:

① プレイヤーが来店 → スタッフが空席を案内 or プレイヤーが空席に着席
↓
② Floorアプリ / Manager座席表に「新着席（受付未処理）」が表示
↓
③ スタッフがテーブルに出向き → QRスキャン → 本人確認 → アラート確認
↓
④ バイイン額確認 → 支払い処理 → チップ配布
↓
⑤ 座席表を「プレイイン」に更新

新規来店（アカウントなし）の場合: テーブルでゲスト登録（B3-1-03）を実施してからテーブル受付に進む。Floorアプリでゲスト登録 → QR発行 → そのままテーブル受付の流れで、プレイヤーを席から移動させずに対応可能。

8. 検討事項・リスク

セクション項目内容対応方針
CQRコードのセキュリティQRコードのスクリーンショットによるなりすまし防止QRコードに有効期限を設定（30秒〜1分で自動更新）。ワンタイムトークン方式を検討
C受付混雑時のスループット大規模トーナメント（100名以上）の開場時に受付が集中セルフチェックイン端末（店舗設置QR方式）の活用。複数スタッフによるFloorアプリでの並列処理
Cオフライン時の対応店舗のネットワーク障害時のチェックイン/テーブル受付処理緊急モード（手動記録 → オンライン復旧後に一括同期）を検討
T出入り禁止プレイヤーの事前エントリー出入り禁止フラグがオンラインエントリー後に設定される場合QRスキャン時にアラートを表示し、チェックインをブロック。返金はキャンセルポリシーに従う
Tノーショーの繰り返し常習的にノーショーするプレイヤーへの対策ノーショー回数の記録。一定回数超過時にアラート表示。将来的にはセルフレジストレーション制限を検討
R着席前のプレイヤー確認着席してからQRスキャンする場合、不正プレイヤーが一時的に席を占有する可能性顧客アラート（B3-2-04）によりスキャン時にブロック。席の仮占有は短時間で発生リスクは低い
Rテーブル受付のスタッフ通知プレイヤーが着席したがスタッフが気付かない場合Floorアプリに「新着席プレイヤーあり」通知を表示。席QRスキャンによりセルフ受付でカバー
Rテーブルでの決済手段制限テーブルサイドではカード決済端末が手元にない場合があるポータブル決済端末の導入、またはデポジット方式（事前に与信確保）での対応。現金は問題なし
