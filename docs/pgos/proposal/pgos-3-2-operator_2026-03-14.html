1. このシチュエーションの概要

店舗の課題

「当日の参加者数が開始まで読めず、テーブル配置が直前まで決まらない。リングゲームのウェイティングも紙で管理していて、連絡も個別に電話やLINE。事前に参加者を把握でき、効率的にウェイティングを管理したい。」

PGOSでは、プレイヤーのセルフレジストレーション（事前エントリー）をリアルタイムで受付・管理。事前エントリー数を把握して準備に活かせる。ウェイティングリストはアプリからの登録・自動通知に対応し、スタッフの連絡業務を削減。受付期間の設定と受付終了後のスタッフ操作の柔軟性を両立。

2. ユーザーストーリー

ストーリー1: 「事前エントリーを活用してスムーズに開始」

店舗スタッフAは、本日19時開始の「デイリーNLH」のPGOS Managerの参加者管理画面を確認。15名が事前エントリー（セルフレジストレーション）済み。うち3名はポイント決済完了、12名は現金（当日店頭払い）。事前エントリー数から2テーブルを準備。開始30分前に受付を開始し、当日参加の4名をカウンターで登録。チェックインとエントリーを同時処理し、即座にテーブル抽選を実行。

ストーリー2: 「受付終了後の遅刻者をスタッフ操作で対応」

トーナメント開始後、遅刻したプレイヤーBが来店。受付は既に終了しているが、テーブルにはBのためのチップが置かれている。スタッフはPGOS Managerで手動エントリーを実行。「受付終了後のエントリーです。続行しますか？」のアラートが表示されるが、「続行」を選択してBをチェックインとエントリーの同時処理で登録。

ストーリー3: 「ウェイティングリストの自動管理」

金曜夜、NLH 1-2テーブルが満席。アプリからウェイティング登録したプレイヤーが5名。プレイヤーCが離席すると、ウェイティングリストの先頭プレイヤーに自動でプッシュ通知が送信。スタッフAはManagerの画面で通知状況を確認。10分経過しても反応がない場合は自動スキップされ、次のプレイヤーに通知。

3. プレイヤーのトーナメント状態モデル（復習）

状態英語意味Manager 上の表示3-2 での関連
レジストレーション済Registeredエントリー登録済み。チェックイン前「登録済み」（チェックイン待ち）セルフレジストレーションで到達する状態
プレイインPlay-inチェックイン完了。テーブルに着席「着席中」（テーブル/シート表示）チェックイン処理後に遷移
シートアウトSeat-outバスト。受付中なら保留（リエントリ可）「バスト」（受付中は「リエントリ待ち」）リエントリ管理に関連
プレイアウトPlay-outトーナメントから完全退出「退出」（順位確定）—

運営側のポイント:

  • バスト時、受付中（レジストレーションオープン）であればシートアウト状態で保留。リエントリ（再プレイイン）の可能性がある
  • レジストレーションクローズ後のバストでは、シートアウトとプレイアウトが同時に発生し、順位が確定
  • スタッフは「チェックイン + エントリー同時処理」（B1-3-09）で、レジストレーション済 → プレイインへ直接遷移させることができる

4. トーナメントの受付状態と制御

受付状態トリガーセルフレジストレーション（アプリ）スタッフ操作（Manager）
受付前トーナメント作成時（デフォルト）不可不可
受付中クラブ指定時刻で自動遷移、またはスタッフ手動変更可能可能
受付終了クラブ設定のクローズ時刻で自動遷移、またはスタッフ手動変更不可（ブロック）可能（アラート付き）
進行中トーナメント開始不可可能（アラート付き）
終了全プレイヤーがプレイアウト、またはスタッフ手動終了不可不可

受付開始タイミングの設定方法

  • 絶対指定: 「2026年3月15日 17:00に受付開始」のような日時指定
  • 相対指定: 「トーナメント開始の2時間前に受付開始」のような相対設定
  • 手動: スタッフがPGOS Managerでステータスを「受付中」に変更

セルフレジストレーション（アプリからのエントリー）は「受付中」の間のみ可能。受付終了後は、スタッフ操作のみでエントリーを登録できる（テーブルにチップを置いて後から来る等のケースに対応）。

6. セルフレジストレーション受付の運用フロー

シナリオ A: 事前エントリーの多いトーナメント

  1. トーナメント作成時にレジストレーション期間を設定（開始2時間前 〜 レベル8終了時）
  2. 受付開始 → プレイヤーアプリに「エントリーする」ボタンが表示される
  3. セルフレジストレーション受付 → 参加者管理画面に「セルフ」としてリアルタイム追加
  4. 事前エントリー数を確認してテーブル数を決定（例: 18名 → 2テーブル）
  5. 当日来店プレイヤーをカウンターで登録（チェックイン + エントリー同時処理）
  6. テーブル・シートのランダム抽選を実行
  7. トーナメント開始 → 受付終了時刻でセルフレジストレーション自動ブロック
  8. 遅刻者はスタッフ操作でエントリー（アラート確認後に続行）

シナリオ B: 当日受付のみのトーナメント

  1. セルフレジストレーションを無効に設定（B1-3-05で受付開始を設定しない、または手動のみ）
  2. 来店プレイヤーを受付でスタッフが登録（チェックイン + エントリー同時処理）
  3. ゲスト登録（アカウント未保有者）にも対応
  4. スタッフが手動で受付終了に変更

7. ウェイティングリスト運用フロー

基本フロー（スタッフ視点）:

① テーブルオープン → テーブル情報がプレイヤーアプリに公開
↓
② プレイヤーがアプリからウェイティング登録 → Manager のウェイティングリストにリアルタイム追加
  または、店頭でスタッフがウェイティング登録（同一キュー）
↓
③ テーブルに空席発生（プレイヤーが離席）
↓ 自動トリガー
④ ウェイティングリスト先頭のプレイヤーにプッシュ通知を自動送信
↓
⑤-a プレイヤーが来店 → チェックイン → チップイン → 着席
⑤-b 10分間無応答 → 自動スキップ → 次のプレイヤーに通知
⑤-c プレイヤーが「あとで」操作 → 順番を後ろに移動

無応答時間の設定

設定項目デフォルト値備考
順番到来後の応答期限10分店舗ごとに変更可能
「あと○人」の事前通知あと1人通知タイミングを店舗が設定
スキップ後の扱いリストから削除「後ろに回す」設定も可能

9. 検討事項・リスク

項目内容対応方針
セルフレジストレーションの無断不参加（No-show）事前エントリー後に来店しないプレイヤーへの対応。テーブル配置の計画が狂う店舗がキャンセルポリシーを設定可能に（無料キャンセル期限、キャンセル料/ポイントペナルティ）。No-show 率の記録・レポート
ゲストプレイヤーの管理アカウント未保有者の参加管理。戦績の紐付けが困難ゲスト登録で最低限の参加管理を提供。正式登録への導線を設置。後日アカウント作成時にゲスト戦績のマージ（Could）
受付終了後のスタッフ操作の記録受付終了後のエントリーが頻発する場合、受付期間設定が適切でない可能性受付終了後エントリーの頻度をダッシュボードで表示。受付終了時刻の見直しを提案
ポイント決済と現金の混在同一トーナメントでポイント決済と現金決済が混在する場合の精算管理支払い方法を参加者ごとに記録し、精算時にポイント利用分を自動集計
ウェイティングのアプリ依存アプリを持たない来店客のウェイティング管理スタッフがManagerから手動でウェイティング登録。アプリ登録と同一キューで管理
