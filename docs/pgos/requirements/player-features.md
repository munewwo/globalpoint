## A1. 店舗・イベント発見

プレイヤーが「今日どこでポーカーを遊べるか」を素早く見つけられること。
現行の POKERGUILD スケジュールサイトと Waitinglist の店舗検索を統合・強化する。

### A1-2. イベント・スケジュール検索

#### プレイヤーのトーナメント状態モデル

プレイヤーとトーナメントの関係は以下の状態遷移で管理する。表示名の混乱を避けるため、各状態を明確に定義する。

| 状態 | 英語表記 | 意味 | プレイヤーアプリでの表示 |
|------|---------|------|----------------------|
| レジストレーション済 | Registered | エントリー登録済みだが、まだチェックイン（着席）していない | 「登録済み — チェックイン待ち」 |
| オルタネイト | Alternate | チェックイン済みだが自動配席できる席がない状態。スタッフが順番に呼び出し配席する | 「オルタネイト — スタッフが席を案内します」 |
| プレイイン | Play-in | テーブルに着席してプレイ中 | 「プレイ中」（リアルタイム情報表示） |
| シートアウト（バスト） | Seat-out (Bust) | バスト（チップ全損）またはセルフバスト（自発的申告）によりテーブルから離れた状態。レジストレーションクローズ前であれば保留され、リエントリ（再プレイイン）の可能性がある | 「バスト — リエントリ可能」（受付中の場合）|
| プレイアウト | Play-out | トーナメントから完全に退出した状態。仮順位確定。受付中のセルフチェックアウト（自発的プレイアウト）は不可 | 「終了」（結果表示） |

```
レジストレーション済 → [チェックイン] → オルタネイト（席なし時）→ [スタッフ配席] → プレイイン
                                      → プレイイン（席あり時 = 自動配席）
                                                                     ↓ [バスト / セルフバスト]
                                                                シートアウト
                                                                ├─ 受付中 → [リエントリ] → プレイイン
                                                                └─ 受付終了 → プレイアウト（仮順位確定）
```

#### トーナメントの受付状態モデル

| 状態 | 意味 | プレイヤーアプリ側 | クラブ管理側 |
|------|------|------------------|------------|
| 受付前 | トーナメント作成済みだが受付開始前 | 「受付前」表示。エントリーボタン非表示 | — |
| 受付中 | エントリー受付中 | セルフレジストレーション可能。締切カウントダウン表示 | エントリー登録可能 |
| 受付終了 | レジストレーションクローズ | セルフレジストレーション不可。「受付終了」表示 | スタッフ操作でのエントリー登録は引き続き可能（アラート表示付き）。締切後の来店者対応等に対応 |
| 進行中 | トーナメントプレイ中 | プレイイン中のプレイヤーにリアルタイム情報表示 | — |
| 終了 | トーナメント完了 | 結果表示 | ペイアウト処理 |

受付開始のトリガー:
- クラブがあらかじめ指定した時刻（例: トーナメント開始1時間前）で自動的に受付中に遷移、または
- スタッフがPGOS Managerでトーナメントステータスを手動で「受付中」に変更

受付終了のトリガー:
- クラブが設定したレジストレーションクローズ時刻で自動遷移、または
- スタッフが手動で「受付終了」に変更

**セルフ vs スタッフ操作の分離**: 受付終了後、プレイヤーアプリからのセルフレジストレーションはシステム的にブロックされる。一方、スタッフによるPGOS Manager上での登録操作は時間外でも可能（アラート「受付終了後のエントリーです。続行しますか？」を表示）。テーブルにチップを置いて後から来店する想定に対応。

#### 大会・シリーズの階層構造

トーナメントのグルーピングは以下の3階層で管理する。

| 階層 | 定義 | 例 |
|------|------|---|
| **シリーズ** | 複数の大会をまとめるグループ | JAPAN POKER TOUR 2026 |
| **大会** | イベント型クラブが管理するトーナメントグループ。実施場所・期間の情報を持つ | JPT 東京大会（3/20-3/25、ホテルX） |
| **トーナメント** | 個別のトーナメント | NLH メインイベント、PLO サイドイベント |

- 大会はイベント型クラブで作成・管理可能。店舗型クラブが直接主催するトーナメントは大会に属さない（ハウストーナメント）
- 店舗型クラブが大会に参加する場合（サテライト開催等）は、大会主催のイベント型クラブからの紐付けで対応
- シリーズ > 大会 > トーナメント の構造により、大規模イベントの一覧性と個別トーナメントの発見性を両立

| ID | 要件 | 優先度 |
|----|------|--------|
| A1-2-01 | トーナメントとリングゲームを分けて表示する。ホーム画面・検索結果ともにセクション分離し、それぞれの文脈に適した情報表示を行う | Must |
| A1-2-02 | 日付・エリア・ゲーム種別・エントリー費用帯・タグでフィルタ | Must |
| A1-2-03 | トーナメント詳細: ストラクチャー・エントリー費・ギャランティー・開始時刻 | Must |

| A1-2-05 | 大会・シリーズとハウストーナメントの区分表示: シリーズ＞大会＞トーナメントの階層構造で表示。大会はまとまった情報ページ（実施場所・期間・トーナメント一覧・総GTD）を持ち、ハウストーナメントとは専用セクションまたはタブで分けて表示する。統合検索でも発見できるようフィルタ対応 | Should |
| A1-2-06 | iCal / Google Calendar 連携（イベントの外部カレンダー登録） | Could |

| A1-2-09 | ホットトーナメント表示: エントリー数が定員の70%以上に達したトーナメントに🔥アイコンを表示。「人気」「残りわずか」等のバッジで視覚的に強調。ただしクラブがエントリー数を非公開設定（B1-3-08）にしている場合は、🔥バッジもエントリー数も非表示 | Should |
| A1-2-10 | バリュー指標表示: プライズプール（GTD含む）÷ エントリー費のオッズ（バリュー）を算出し表示。バリューが高いトーナメントには「HIGH VALUE」バッジを付与（例: GTD ¥100,000 / エントリー ¥3,000 → 33.3x）。※景品表示による集客が風営法上の問題となる可能性があるため、コンプライアンス確認の結果によっては本機能を提供しない判断もありうる | Should |
| A1-2-11 | エントリー数予想・オッズ予想の表示: 過去の同種トーナメントの実績データに基づき、予想エントリー数と期待値を表示（※風営法上の表示リスクを要検討。表示可否は法務確認後に決定。非表示の場合はデータを内部分析用に留める） | Could |

## A2. プレイ体験

店舗での遊技をスムーズにする機能群。来店前から来店中までをカバーする。
トーナメント（A2-T）とリングゲーム（A2-R）に分けて要件を定義する。

### A2-0. 共通設定

| ID | 要件 | 優先度 |
|----|------|--------|
| A2-0-01 | 匿名モード: プレイヤーがトーナメント参加者リスト上での表示名を非表示（GuestXXXXX）にできるアカウント全体の設定。他のプレイヤーから参加中のトーナメントを知られたくない場合に利用。ただしランキング（A3-4）、席移動・テーブルバランシングのシステム通知、PGOS Clock（店内TV表示）、PGOS Manager（スタッフ画面）、レビュー投稿では実名表示が維持される | Should |
| A2-0-02 | 支払い方法の選択: クレジットカード（セルフレジストレーション時のオンライン決済）/ クレジットカード（店頭端末決済）/ 現金 / ウェブコイン（PGOSポイント）/ クラブポイント（各クラブが独自に発行する店舗限定ポイント）/ チケット（サテライト通過等で獲得）から選択。クラブポイントの残高表示は風営法対策として店舗チェックイン中に限定する可能性あり | Must |
| A2-0-03 | 支払いタイミング: ① 事前オンライン（セルフレジストレーション時にオンライン決済）、② 都度前払い（来店時に受付で支払い）、③ 退店時まとめ払い（後払い）の3種類。クラブごとに対応する支払いタイミングを設定 | Must |
| A2-0-04 | 後払い用デポジット: クレジットカードの与信枠確保（オーソリゼーション）によるデポジット機能。最大与信額は10万円。アカウントごとにデポジット額を管理。退店時に実売上として確定（キャプチャ）。デポジット決済はアカウントの履歴に表示され、即時返金（決済キャンセル / オーソリ取消）が可能 | Should |
| A2-0-05 | チケット・クーポンからの逆引き検索: 取得済みのチケットやクーポンから、利用可能なトーナメント・リングゲームの一覧を表示。チケット詳細画面から対象イベントへ直接遷移可能 | Should |
| A2-0-06 | チケット利用順序: 同一トーナメントに利用可能なチケットが複数ある場合、有効期限が短いものから自動選択。プレイヤーが手動で変更することも可能 | Must |

### A2-T. トーナメント参加

トーナメントのエントリー（セルフレジストレーション）からチェックアウト（プレイアウト）までの要件。

#### トーナメント状態モデル（拡張版）

| 状態 | 英語 | 意味 | アプリ表示 |
|------|------|------|----------|
| レジストレーション済 | Registered | エントリー登録済み、チェックイン前 | 「登録済み — チェックイン待ち」 |
| オルタネイト | Alternate | チェックイン済みだが自動配席できる席がない状態。スタッフが順番に呼び出し配席 | 「オルタネイト — スタッフが席を案内します」 |
| プレイイン | Play-in | テーブルに着席してプレイ中 | 「プレイ中」 |
| シートアウト | Seat-out | バスト。受付中なら保留（リエントリ可） | 「バスト — リエントリ可能」 |
| プレイアウト | Play-out | トーナメントから完全退出。仮順位確定 | 「終了」 |

```
レジストレーション済 → [チェックイン] → オルタネイト（席なし時）
                                         ↓ [スタッフ配席]
                                     プレイイン
                                         ↓ [バスト]
                                    シートアウト
                                    ├─ 受付中 → [リエントリ] → プレイイン
                                    └─ 受付終了 → プレイアウト（仮順位確定）

レジストレーション済 → [チェックイン] → プレイイン（席あり時 = 自動配席）
```

- **セルフバスト**: 受付中に限り、プレイヤーが自発的にバスト（シートアウト）を申告可能。リエントリへの導線
- **セルフチェックアウト**: 受付中はセルフチェックアウト（自発的プレイアウト）不可。仮順位の操作を防止するため
- **オルタネイト → 自動配席の切り替え**: オルタネイト状態から自動配席への切り替えは自動では行わない。スタッフが任意で自動配席モードに切り替え可能

| ID | 要件 | 優先度 |
|----|------|--------|
| A2-T-01 | セルフレジストレーション: アプリからトーナメントへの事前エントリー。受付中の場合のみ利用可能。エントリー確定後「レジストレーション済」に遷移 | Must |
| A2-T-02 | 支払い方法の選択と決済: エントリー費用の確認。支払い方法（A2-0-02）から選択。クーポン適用可（クーポン割引後の金額にポイント/ウェブコインを適用）。併用不可クーポンは1枚のみ選択可。手数料の事前表示 | Must |
| A2-T-03 | チケットによるエントリー: サテライト通過等で獲得したチケットでエントリー。チケットは自動保存され、対象トーナメントのエントリー時に「チケットを使用」オプション表示。複数チケットがある場合は有効期限が短いものを自動選択 | Must |

#### ショットクロック（チェスクロック方式）

Triton Poker Series の Triton Tempo に着想を得たシステム。各プレイヤーにトーナメント全体を通した個人タイムバンクを付与し、ベースタイム超過分を秒単位で消費する。従来のタイムエクステンションチップ（30秒単位）と異なり、必要な時間だけ使えるため公平性が高く、ゲーム進行速度も向上する。

#### リングゲームの着席優先モデル

リングゲームでは「先に着席、後から手続き」のフローを基本とする。プレイヤーはウェイティングリストで呼ばれた後（またはウォークインで空席がある場合）、まずテーブルに着席する。着席後、スタッフがテーブルに出向いて本人確認・バイイン手続きを行うか、プレイヤーが席QRをスキャンしてセルフ手続きを行う。トーナメントのように全員が受付カウンターを通過する必要はなく、テーブルで手続きが完結する設計。

## ユーザーストーリー（主要シナリオ）

### シナリオ 1: 「今日ポーカーがしたい」

> プレイヤーA は仕事帰りにポーカーを遊びたい。PGOS Player アプリを開き、現在地周辺の店舗を検索。NLH 1-2 のリングゲームが開催中の店舗を見つけ、ウェイティングリストに登録。電車で移動中に「あと2人です」の通知を受信。店舗到着後、QRコードでチェックイン。プレイ終了後、プレイ時間が自動記録される。

### シナリオ 2: 「週末のトーナメントに参加したい」

> プレイヤーB はカレンダーで週末のトーナメントを検索。エントリー費 ¥5,000 + ウェブコイン利用可能なイベントを発見。アプリから事前エントリーし、ポイントで支払い。当日会場でチェックインし、トーナメントに参加。3位入賞し、選手契約済みのためポイントが自動付与される。結果はプロフィールの戦績に反映。

### シナリオ 3: 「自分のランキングを確認したい」

> プレイヤーC はプロフィールページを開き、今月のトーナメント成績を確認。地域別ランキングで現在15位であることを確認。フレンドのプレイヤーD が同じイベントで入賞したことをフィードで知り、「いいね」を送る。
