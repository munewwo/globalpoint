### B1-3. 参加者管理

#### プレイヤーのトーナメント状態モデル

プレイヤーとトーナメントの関係は以下の状態で管理する（player-features.md A1-2 と対応）。


| 状態          | 英語表記       | 意味                                            | Manager上の表示           |
| ----------- | ---------- | --------------------------------------------- | --------------------- |
| レジストレーション済  | Registered | エントリー登録済み、チェックイン（着席）前                         | 「登録済み」（チェックイン待ち）      |
| オルタネイト      | Alternate  | チェックイン済みだが自動配席できる席がない状態。スタッフが順番に呼び出し配席        | 「オルタネイト」（配席待ち順リスト）    |
| プレイイン       | Play-in    | テーブルに着席してプレイ中                                 | 「着席中」（テーブル/シート表示）     |
| シートアウト（バスト） | Seat-out   | バストまたはセルフバストによりテーブルから離れた状態。受付中であれば保留（リエントリ可能） | 「バスト」（受付中なら「リエントリ待ち」） |
| プレイアウト      | Play-out   | トーナメントから完全に退出。仮順位確定                           | 「退出」（順位確定）            |


- バスト時、受付中（レジストレーションオープン）であればシートアウト状態で保留。リエントリ（再プレイイン）の可能性がある
- レジストレーションクローズ後のバストでは、シートアウトとプレイアウトが同時に発生し、順位が確定する
- オルタネイト: 全テーブル満席、またはバランシングの都合で即座に配席できない場合に遷移。システムは自動的にオルタネイト状態に切り替え可能だが、オルタネイトから自動配席への復帰は自動で行わない（スタッフが任意で切り替え）
- 受付中のセルフチェックアウト（自発的プレイアウト）はプレイヤー側からは不可。仮順位操作の防止

#### トーナメントの受付状態と制御


| 状態   | トリガー                              | セルフレジストレーション（プレイヤーアプリ） | スタッフ操作（PGOS Manager）                   |
| ---- | --------------------------------- | ---------------------- | -------------------------------------- |
| 受付前  | トーナメント作成時（デフォルト）                  | 不可                     | 不可                                     |
| 受付中  | クラブが指定した時刻で自動遷移、またはスタッフが手動で変更     | 可能                     | 可能                                     |
| 受付終了 | クラブが設定したクローズ時刻で自動遷移、またはスタッフが手動で変更 | **不可**（システム的にブロック）     | **可能**（アラート「受付終了後のエントリーです。続行しますか？」を表示） |
| 進行中  | トーナメント開始                          | 不可                     | 上記アラート付きで可能（受付終了後と同様）                  |
| 終了   | 全プレイヤーがプレイアウト、またはスタッフが手動で終了       | 不可                     | 不可                                     |


- 受付開始タイミングは柔軟に設定可能: 「トーナメント開始の○時間前」のような相対指定、または具体的な日時の絶対指定
- セルフ vs スタッフの分離により、システム的な締切管理と現場の柔軟性を両立


| ID      | 要件                                                                                                                                                                      | 優先度    |
| ------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ |

### B1-6. マルチデイ・大会・シリーズ

トーナメントは以下の3階層でグルーピングできる。


| 階層         | 定義                   | 作成主体               | 例                        |
| ---------- | -------------------- | ------------------ | ------------------------ |
| **シリーズ**   | 複数の大会をまとめるグループ       | PGOS管理者 / イベント型クラブ | JAPAN POKER TOUR 2026    |
| **大会**     | 実施場所・期間を持つトーナメントグループ | イベント型クラブ           | JPT 東京大会（3/20-3/25、ホテルX） |
| **トーナメント** | 個別のトーナメント            | 店舗型 / イベント型クラブ     | NLH メインイベント              |


- 店舗型クラブが直接主催するトーナメントは大会に属さない（ハウストーナメント）
- イベント型クラブが大会を作成し、その中に複数のトーナメントをグルーピングする
- 店舗型クラブが大会に参加する場合（サテライト開催等）は、大会主催のイベント型クラブからの紐付けで対応
- 大会には実施場所（会場名・住所）と期間（開始日〜終了日）の情報を持たせる

| ------- | ----------------------------------------------------------------------------------------------- | ------ |

| B1-6-04 | シリーズの作成・管理: 複数の大会をまとめてシリーズを構成。シリーズ詳細ページ（大会一覧・全体スケジュール）を自動生成                                     | Should |

## ユーザーストーリー（主要シナリオ）

### シナリオ 1: 「今日のトーナメントを運営する」

> 店舗スタッフA は PGOS Manager にログインし、本日のデイリートーナメントを開始。事前エントリー済みの12名を確認し、当日参加者を追加登録。テーブル・シートを自動抽選で割り当て。店内のTVに PGOS Clock を表示し、レベル進行を開始。レベル変更やブレイクの通知が自動で流れる。参加者の脱落を記録し、最終的に8名が入賞。ペイアウトを自動計算し、ポイント付与リクエストを送信。結果はプレイヤーの戦績に自動反映される。

### シナリオ 2: 「リングゲームを回す」

> 店舗スタッフB は NLH 1-2 テーブルをオープン。アプリからのウェイティング登録が3件入っていることを確認し、順番に案内。プレイヤーがチップインする際に金額を記録。プレイヤーが離席する際にチップアウトを記録。全セッションのチップ増減・プレイ時間が自動で記録される。

### シナリオ 3: 「月次の経営状況を確認する」

> 店舗オーナーC は月次レポートを確認。トーナメント収入・リングゲーム収入・飲食収入の内訳と、ポイント利用分の精算状況を確認。前月比で来店者数が15%増加していることを確認し、新しいトーナメントストラクチャーの導入が効果的だったと判断。
