# PGOS 要件定義 — 店舗・イベント運営側 機能要件

## 対象コンポーネント

- **PGOS Manager**（Webアプリ / タブレット最適化）
- **PGOS Clock**（トーナメントタイマー / クロック — スマートTV / Fire TV / ブラウザ）

---

## B1. トーナメント管理

現行 POKERWEB のトーナメント機能を置き換えつつ、Poker Fans のタイマー同期と同等以上の体験を実現する。

### B1-1. ストラクチャー管理

| ID | 要件 | 優先度 |
|----|------|--------|
| B1-1-01 | ブラインドレベル・アンティ・時間を設定したストラクチャーを作成できる | Must |
| B1-1-02 | ストラクチャーのテンプレート保存・再利用 | Must |
| B1-1-03 | ストラクチャーテンプレートのインポート/エクスポート（JSON / CSV） | Should |
| B1-1-04 | ストラクチャーの公開テンプレートライブラリ（他店舗のテンプレートを参照・コピー） | Could |

| B1-1-06 | ブレイク（休憩）の自動挿入設定 | Must |

## B3. 会員管理 (CRM)

プレイヤーの情報と来店履歴を管理し、店舗のマーケティング活動を支援する。

### B3-1. 会員登録・認識

| ID | 要件 | 優先度 |
|----|------|--------|
| B3-1-01 | PGOS Player アカウントとの紐付けによる会員登録 | Must |
| B3-1-02 | QRコードスキャンによる来店時の会員認識 | Must |
| B3-1-03 | PGOS 未登録者のゲスト登録（氏名・連絡先の手動入力） | Must |
| B3-1-04 | ゲスト → PGOS 会員への移行促進（QRコード発行等） | Should |

### B3-2. 会員情報管理

| ID | 要件 | 優先度 |
|----|------|--------|
| B3-2-01 | 会員一覧の表示・検索（名前・ID・来店日等） | Must |
| B3-2-02 | 会員種別・グループの設定（VIP / 常連 / 新規 / 注意等） | Must |
| B3-2-03 | 会員ごとのメモ・ノート（店舗スタッフ間で共有） | Should |
| B3-2-04 | 顧客アラート（注意人物のチェックイン時に警告表示） | Must |
| B3-2-05 | eKYC ステータスの確認 | Must |
| B3-2-06 | クラブに共有される会員情報のスコープ管理（下表参照） | Must |

**クラブへの会員情報共有ポリシー:**

| 情報 | 共有条件 | 備考 |
|------|---------|------|
| ユーザーID（表示名） | 常時 | システム上必須 |
| 年代タグ | 常時 | 風営法対応に必須（正確な生年月日は共有しない） |
| eKYCステータス | 常時 | ポイント利用判定に必須 |
| 氏名（実名） | eKYC完了後 | 本人確認済みの場合のみ |
| メールアドレス | 正式登録後 | CRM・連絡用（プレイヤーが初回チェックイン時に同意） |
| 電話番号 | SMS認証後 | 緊急連絡用（プレイヤーがオプトインした場合のみ） |
| 生年月日 | 共有しない | 年代タグで代替。正確な日付はシステム側のみ管理 |
| 国 | 正式登録後 | 統計用 |

- プレイヤーはクラブへの初回登録時（初回チェックイン時）に「このクラブに以下の情報を共有します」と明示され、同意する
- 「次回から表示しない」チェックボックスにより、2回目以降の別クラブ登録時は同意画面を省略可能
- 電話番号の共有はオプトイン（プレイヤー設定で制御可能）
- クラブからプレイヤーへの連絡は原則としてPGOSアプリ内メッセージング（B3-4）で完結する設計とし、個人連絡先の直接共有を最小化する

## ユーザーストーリー（主要シナリオ）

### シナリオ 1: 「今日のトーナメントを運営する」

> 店舗スタッフA は PGOS Manager にログインし、本日のデイリートーナメントを開始。事前エントリー済みの12名を確認し、当日参加者を追加登録。テーブル・シートを自動抽選で割り当て。店内のTVに PGOS Clock を表示し、レベル進行を開始。レベル変更やブレイクの通知が自動で流れる。参加者の脱落を記録し、最終的に8名が入賞。ペイアウトを自動計算し、ポイント付与リクエストを送信。結果はプレイヤーの戦績に自動反映される。

### シナリオ 2: 「リングゲームを回す」

> 店舗スタッフB は NLH 1-2 テーブルをオープン。アプリからのウェイティング登録が3件入っていることを確認し、順番に案内。プレイヤーがチップインする際に金額を記録。プレイヤーが離席する際にチップアウトを記録。全セッションのチップ増減・プレイ時間が自動で記録される。

### シナリオ 3: 「月次の経営状況を確認する」

> 店舗オーナーC は月次レポートを確認。トーナメント収入・リングゲーム収入・飲食収入の内訳と、ポイント利用分の精算状況を確認。前月比で来店者数が15%増加していることを確認し、新しいトーナメントストラクチャーの導入が効果的だったと判断。

## 非機能要件（店舗側固有）

| 項目 | 要件 |
|------|------|
| 画面応答時間 | 管理画面の操作レスポンスは2秒以内 |
| 同時接続 | プランに応じた同時ログイン数を保証 |
| タイマー精度 | クロックのカウントダウン誤差は1秒以内 |
| タイマー同期 | マルチデバイス間の同期ずれは2秒以内 |
| オフライン | トーナメントクロックはネットワーク切断時も動作を継続 |
| データ保存 | トーナメント・リングゲームの運営データは最低5年間保存 |
| ブラウザ対応 | Chrome / Safari / Edge 最新2バージョン |
